# План разработки WB AI-агента

**Для:** Муслим Магомаев
**Цель:** AI-агент для управления внутренней рекламой на Wildberries
**Срок до MVP:** 7-9 недель при 1-2 часах в день
**Хостинг:** Railway (на старте)
**Стек:** Python + Claude API + Telegram Bot API + PostgreSQL + n8n (как опциональный мост)

---

## Архитектура в одной картинке

```
[WB Advert API]  ←→  [Python-сервис на Railway]  ←→  [Claude API]
                            ↓        ↑
                     [PostgreSQL]    │
                            ↓        │
                     [Telegram Bot]  │
                            ↓        │
                         [Муслим] ───┘ (подтверждения, обратная связь)
```

Сервис крутится 24/7. По расписанию ходит в WB, тянет данные, прогоняет через правила и Claude, пишет в телегу, ждёт твоих подтверждений, применяет изменения.

## Что ты делаешь руками (без кода)

Это до начала разработки или параллельно с фазой 1. Без этого код запускать некуда.

1. **Договорённость с клиентом**
   - Письменное разрешение использовать его кабинет для разработки
   - Чётко: что бот делает сам, что только предлагает, в каком режиме (тест/бой)
   - Бэкап текущих ставок и кампаний на день старта — на всякий случай

2. **Получить API-токены WB**
   - Зайти в личный кабинет WB → Настройки → Доступ к API
   - Создать токен с правами **«Продвижение»** (для управления РК)
   - Дополнительно: токен **«Статистика»** (для отчётов)
   - Сохранить токены в менеджер паролей (Bitwarden, 1Password). **Никуда больше не вставлять**.
   - **До написания кода** — проверить ключевые методы вручную через Postman или httpx-скрипт: список кампаний, статистика, кластеры. WB API печально известен расхождением между документацией и реальностью.

3. **Зарегистрировать Telegram-бота**
   - Открыть [@BotFather](https://t.me/BotFather) в телеге
   - `/newbot` → имя → username (например `muslim_wb_agent_bot`)
   - Сохранить токен
   - `/setcommands` для бота — добавишь позже из плана

4. **Получить API-ключ Claude**
   - console.anthropic.com → API Keys → Create Key
   - Пополнить баланс на $10-20 (хватит на 2-3 недели разработки и тестов)

5. **Зарегистрироваться на Railway**
   - railway.com → войти через GitHub
   - Бесплатный план на старте, потом ~$5/мес за основной сервис + ~$5 за БД

6. **Создать GitHub-репозиторий**
   - Приватный, имя например `wb-ai-agent`
   - В нём будет жить весь код
   - Claude Code будет с ним работать

7. **Выгрузить правила из регламента**
   - Открыть свой регламент по WB
   - Выписать в отдельный документ `rules.md`:
     - Правила управления ставками (когда повышать, когда понижать, шаг, потолок, дно)
     - Правила удаления кластеров (по каким метрикам, за какой период)
     - Правила возврата ранее удалённых кластеров
   - Это станет источником истины для кода. Без этого документа не начинай фазу 2.

## Фаза 0 — подготовка окружения (3-4 дня по 1-2 ч)

- В терминале в папке проекта: `claude`

Стек: Python 3.11, FastAPI (минимально), Telegram-бот через aiogram 3.x, PostgreSQL через SQLAlchemy, планировщик APScheduler, Claude API через anthropic SDK. Хостинг Railway. Сделай начальную структуру проекта: requirements.txt, базовый main.py, конфиг через pydantic-settings, подключение к БД, заглушку для health-check эндпоинта. Объясни каждый файл одним абзацем.»

- Дать Claude Code разрешения на чтение/запись файлов
- Запустить локально: `python src/main.py`
- Убедиться что health-check отвечает

### День 4. Telegram-бот — «привет»

Промпт Claude Code:

Railway автоматически даёт HTTPS-URL сервиса — используй его как webhook URL. Команды: /start (приветствие), /ping (отвечает pong). Токен бота — из переменной окружения TELEGRAM_BOT_TOKEN.»

## Фаза 1 — чтение данных из WB (1-1.5 недели)

**Цель:** сервис умеет тянуть из WB кампании, кластеры, статистику. Пока ничего не меняет, только читает.

### Задачи

1. **Изучить документацию WB Advert API**
   - Это делаешь сам, до промптов в Claude Code
   - Ключевые методы, которые понадобятся:
     - получить список кампаний
     - получить статистику по кампании за период
     - получить кластеры/поисковые запросы по кампании
     - получить текущие ставки
     - (на потом, фаза 3) изменить ставку, поставить на паузу

2. **Промпт Claude Code на создание WB-клиента**

> «Создай модуль `src/wb/client.py` — обёртку над Wildberries Advert API. Базовый URL: https://advert-api.wildberries.ru. Авторизация: header `Authorization: <токен>`. Реализуй методы:
> - `get_campaigns()` — список кампаний
> - `get_campaign_stats(campaign_id, date_from, date_to)` — статистика
> - `get_keywords(campaign_id)` — поисковые кластеры с метриками
>
> Используй httpx (async). Rate limit WB API — 1 запрос в секунду: добавь `asyncio.sleep(1)` между запросами, не только retry на 429. Сделай ретраи на 429 и 5xx с exponential backoff. Логируй все запросы и ответы. Покрой тестами с mock-ответами.»

3. **Промпт Claude Code на схему БД**

> «Создай SQLAlchemy-модели. Нужна поддержка нескольких клиентов с первого дня, поэтому:
>
> - `Client` — id, name, wb_token, telegram_chat_id
> - `Campaign` — id, client_id (FK), wb_id, название, тип, статус, добавлена_когда
> - `Keyword` — id, campaign_id (FK), фраза, кластер, последний_статус (active/excluded/pending)
> - `KeywordMetric` — id, keyword_id (FK), дата, показы, клики, заказы, расход, CPO, CTR
> - `Decision` — id, keyword_id (FK), решение (keep/remove/borderline), уверенность, причина (текст от Claude), кто решил (ai/human), статус (pending/applied/failed), истекает_когда (TTL), когда создано
>
> Сделай миграции через Alembic. Объясни как накатывать миграции локально и на Railway.»

4. **Команда для бота: /pull**

> «Добавь команду /pull в бота. По команде сервис идёт в WB, тянет все кампании клиента, сохраняет в БД (upsert), и отвечает в чат: "Подтянул X кампаний, Y кластеров за вчера". Доступ к команде — только для моего telegram_user_id (захардкоди в .env как ADMIN_USER_ID).»

5. **Планировщик**

> «Добавь APScheduler. Задача: каждые 3 часа делать то же, что /pull — тянуть данные из WB и сохранять. Логировать в чат только в случае ошибки.»

**Чек-поинт фазы 1:**
- /pull работает, заполняет БД
- автозапуск каждые 3 часа работает
- видишь в логах Railway что данные приходят
- можешь через `railway run psql` зайти в БД и посмотреть таблицы

## Фаза 2 — классификатор кластеров через Claude (2.5-3 недели)

**Это сердце системы. Здесь LLM реально делает работу.**

> **Важно:** первые результаты классификатора будут 60-65%, не 80%. Это нормально. Запускай с этим, собирай обратную связь, улучшай итерациями. Не трать 3 недели на доводку до запуска.

### Задачи

1. **Описать товар клиента**
   - Этот документ ты пишешь сам (или вместе со мной отдельной сессией)
   - Файл `docs/products/<артикул>.md`:
     - Название, категория WB, целевая аудитория
     - 5-10 примеров явно релевантных запросов
     - 5-10 примеров явно нерелевантных
     - Спорные зоны и правила
     - Конкуренты, чьи бренд-запросы НЕ брать

2. **Промпт-инструкция для Claude API**

> «Создай модуль `src/ai/classifier.py`. Функция `classify_keywords(keywords: list[dict], product_context: str) -> list[dict]`. Внутри — вызов Claude API (модель claude-sonnet-4-6 для качества, на тестах claude-haiku-4-5 для дешевизны).
>
> **Важно: используй tool use (function calling), а не свободный JSON в тексте.** Объяви tool `classify_keyword` с параметрами: decision (enum: relevant/irrelevant/borderline), confidence (float 0..1), reason (str). Это гарантирует структуру без хаков с парсингом. Валидируй результат через pydantic.
>
> Промпт строится так: системный промпт описывает задачу классификации, в user-сообщение вкладывается product_context (из markdown-файла товара) + список кластеров в JSON.
>
> Кэширование: если кластер уже классифицирован для этого товара и решение не было переопределено вручную — не отправляй его в Claude повторно. Используй БД.»

3. **Команда /classify**

> «Добавь команду /classify в бота. По команде сервис берёт все некласифицированные кластеры по последней подтянутой кампании, прогоняет через Claude, сохраняет решения в таблицу Decision. В ответ присылает сводку: "Проверил X кластеров: Y релевантных, Z нерелевантных, W спорных".»

4. **Команда /review**

> «Добавь команду /review. Показывает кластеры с decision=irrelevant или borderline по последней классификации, у которых статус=pending и TTL не истёк. Формат: для каждого кластера сообщение с метриками + кнопки "Удалить", "Оставить", "Спорно". Inline-кнопки через aiogram. По нажатию — сохраняет решение human в БД (поле Decision.who = 'human'), переопределяет AI-решение.»

5. **Сбор обратной связи**

> «Когда я нажимаю "Оставить" на кластер, который AI пометил irrelevant — это сигнал. Создай таблицу `Feedback` (keyword, ai_decision, human_decision, client_id, product_id, дата). Эти примеры в фазе 3 пойдут обратно в промпт как few-shot examples.»

**Чек-поинт фазы 2:**
- бот находит нерелевантные кластеры
- ты подтверждаешь / переопределяешь вручную
- решения копятся в БД
- **на этой стадии бот ещё ничего не удаляет в WB** — он только размечает

## Фаза 3 — применение решений + управление ставками (1.5 недели)

**Цель:** бот реально что-то делает в WB, но только с твоего подтверждения.

### Задачи

1. **Метод удаления кластера**

> «Расширь WB-клиент: метод `exclude_keyword(campaign_id, keyword)` — добавляет фразу в минус-слова кампании через WB API. Метод `restore_keyword(campaign_id, keyword)` — обратное действие.»

2. **Команда /apply с идемпотентностью**

> «Добавь команду /apply. Берёт все решения с Decision.who=human, решение=remove, статус=pending, TTL не истёк. Перед применением — сообщение "Применить X удалений? [Да] [Нет]".
>
> При применении: каждое действие меняет статус Decision на applied или failed — не на основе всего батча, а на каждую запись отдельно. Если упало на 3-м из 10 — 2 уже applied, 7 остаются pending и применятся при следующем запуске. В чат — отчёт: "Удалил X кластеров, ошибок Y".»

3. **TTL для рекомендаций**

> «В планировщике: каждые 3 часа после /pull — обновляй TTL у Decision. Записи, у которых истёк_когда < now() и статус=pending — переводи в статус expired. Они не показываются в /review и не применяются. Логика: данные устарели, рекомендация уже не актуальна. TTL по умолчанию = 24 часа, настраивается через переменную окружения.»

4. **Правила управления ставками (без LLM)**

> «Создай модуль `src/rules/bidding.py`. Функция `calculate_bid_action(keyword_metrics) -> {action: 'increase'|'decrease'|'keep', new_bid: int, reason: str}`.
>
> Логику возьми из файла `docs/rules.md` (приложу). Это чистый Python, без вызовов LLM. Покрой тестами на пограничные случаи.»

5. **Подтверждение изменений ставок**

> «Каждые 3 часа после /pull прогоняй ставки через rules. Если есть рекомендации — отправляй в чат сводку с кнопками: "Применить все", "По одной", "Отклонить". Применённые и отклонённые решения логируй в Decision с соответствующим статусом.»

**Чек-поинт фазы 3:** бот может удалять и менять ставки в WB. Все действия — через твоё подтверждение в телеге. Логи всех действий — в БД с явным статусом каждой записи.

## Фаза 4 — ежедневный отчёт через Claude (3-5 дней)

### Задачи

1. **Сбор данных за день**

> «Каждый день в 09:00 МСК собирай метрики за вчера: расход, заказы, ДРР, топ-5 лучших кластеров, топ-5 худших, что изменилось vs предыдущий день.»

2. **Генерация отчёта через Claude**

> «Передавай эти данные в Claude API с промптом: "Ты — аналитик по рекламе WB. Вот данные за вчера: [JSON]. Сделай краткий отчёт (до 1500 символов) для Telegram: 1) общая оценка дня, 2) что работает хорошо, 3) слабые места и рекомендации, 4) на что обратить внимание сегодня. Стиль — деловой, без воды, для эксперта."»

3. **Доставка в телегу**

> «Отправляй отчёт в чат каждое утро. В конце — кнопка "Углубиться в детали" (открывает мини-меню с разрезами).»

**Чек-поинт фазы 4:** утром получаешь содержательную сводку в телеге.

## Где здесь n8n

**Короткий ответ: на старте — нигде.** Не плоди две системы, пока одна не работает.

**Когда он может пригодиться** (после фазы 5):
- Если захочешь добавить нотификации в другие каналы (email, Slack)
- Если будешь интегрироваться с CRM/гугл-таблицами клиентов
- Если будешь делать дашборд с метриками

Тогда n8n работает как «почтальон» поверх твоего сервиса — берёт данные через эндпоинт твоего FastAPI и раскладывает по местам.

## Стратегические напоминания

1. **Это съест ~80-120 часов твоего времени за 7-9 недель.** Учти это в роадмапе. Если reels и регламент важнее — двигай этот проект медленнее, но не бросай.

2. **Правило 90 дней — это про новые направления, а не про автоматизацию текущего.** Этот проект усиливает основную работу, не отвлекает.

3. **Главный риск — перфекционизм на фазе 2.** Классификатор не будет идеальным с первого дня. Первые результаты будут 60-65% — это нормально. Запускай и улучшай через обратную связь. Не доводи до 95% перед запуском — там бесконечно долго.

4. **Второй по величине риск — WB Advert API.** Он меняется без предупреждения и документация отстаёт. Проверяй методы вручную перед написанием кода, закладывай время на отладку.

5. **Кейс для контента:** каждая фаза = пост или рилс в стиле «как я строил AI-агента для WB». Это контент-актив параллельно с разработкой. Тема broad-audience, релевантна и продавцам, и специалистам.

6. **После MVP — не масштабируй сразу на всех клиентов.** Месяц гоняй на одном кабинете, собирай данные о работе, считай leverage. Только потом — упаковка как услуга или продукт. Схема БД уже готова к этому (client_id везде).
